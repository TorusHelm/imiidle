# Правила Для Modifiers

## Назначение

Этот файл фиксирует правила системы модификаторов и мгновенных эффектов.

Цель:
- разделять modifier, instant effect и source,
- не допускать прямого actor -> actor взаимодействия,
- не допускать event hell,
- не смешивать источник эффекта и сам эффект.

## Термины

### Modifier

`Modifier` — это эффект, который:
- применяется к объекту,
- действует некоторое время,
- влияет на поведение объекта.

Примеры:
- `haste`
- `slow`
- `freeze`
- `value_increase`
- `value_decrease`

### Instant Effect

`Instant Effect` — это действие, которое:
- применяется сразу,
- не сохраняется,
- не имеет длительности.

Примеры:
- `charge`
- `reset_cooldown`
- `add_value_instantly`

Правило:
- instant effects не являются modifiers.

### Source

`Source` — это источник эффекта.

Примеры источников:
- `Totem`
- событие комнаты
- инструмент игрока
- случайное событие

Правила:
- имя эффекта не зависит от источника,
- один и тот же modifier, например `haste`, может приходить из разных source,
- source хранится отдельно от самого эффекта,
- логика modifier не зависит от source,
- source не превращает `haste` в отдельный тип modifier,
- modifier не должен менять тип или смысл из-за source.

## Modifier Model

### ModifierDef

`ModifierDef` хранит данные modifier:
- `type`
- параметры, например `multiplier`
- `duration`

### ModifierInstance

`ModifierInstance` хранит runtime-состояние:
- ссылка на `ModifierDef`
- `target`
- `source`
- `remaining_time`

## Правила Наложения

На текущем этапе действует упрощенное правило:
- modifiers не стакуются по силе,
- повторное применение одинакового modifier не суммирует мощность,
- повторное применение обновляет длительность.

Пример:
- `haste x2 for 1s`
- снова `haste x2`
- итог: остается `x2`, но время обновляется.

## Где Хранятся Modifiers

Правила хранения:
- каждый `Plant` хранит список своих активных modifiers,
- `Totem` может хранить modifiers, если они применимы к его модели,
- `Slot` не хранит modifiers,
- `Shelf` не хранит modifiers как состояние конкретных объектов.

Уточнение для `Totem`:
- не каждый modifier обязан влиять на `Totem`,
- так как `Totem` не является time-based actor, часть modifiers может быть для него невалидной или игнорироваться.

## Как Modifiers Влияют На Объект

Объект, например `Plant` или `Totem`, не мутирует базовые значения навсегда.

Вместо этого объект:
- хранит список активных modifiers,
- при расчете выводит итоговый `speed_multiplier`,
- при расчете выводит итоговый `value_multiplier`,
- проверяет состояния вроде `freeze`, `disable` и аналогичные флаги.

Правило:
- modifiers влияют на расчет поведения, а не переписывают permanently base stats.

Итоговое поведение объекта рассчитывается из нескольких слоев:
- базовые параметры объекта,
- timed modifiers,
- aura-влияния.

## Aura

`Aura` — это отдельный вид влияния.

Правила:
- aura не является timed modifier,
- aura не является instant effect,
- aura не имеет `duration` как modifier,
- aura не переапплаится событием каждый tick,
- aura не должна спамить очередь событий.

Архитектурное правило:
- aura — это постоянное правило влияния, пока активен ее source,
- `Shelf` учитывает активные aura-влияния как часть контекста симуляции,
- aura не хранится на цели как обычный active modifier.

## Target Filtering

Не каждый modifier обязан применяться к каждой цели.

Правила:
- source, например `Totem`, задает только target rule,
- `Shelf` выбирает кандидатов и конкретные цели,
- объект сам может решать, принимает ли он modifier,
- объект сам может игнорировать неприменимый modifier.

Примеры:
- `slow` может не применяться к `Totem`, если он не time-based,
- `freeze` может блокировать активацию и у `Plant`, и у `Totem`.

Разделение ответственности:
- source задает правило выбора целей,
- `Shelf` выполняет target selection, включая deterministic и random выбор,
- цель валидирует применимость modifier к себе.

## Flow Применения

Главное правило:
- actor не применяет эффект напрямую другому actor.

Нельзя:
- `Metronome` напрямую вызывает `PlantB.apply_modifier()`.

Правильно:
1. `PlantA` активируется.
2. `PlantA` сообщает об этом в `Shelf`.
3. `Shelf` рассылает локальное событие внутри своей полки.
4. `Totem` получает это событие.
5. `Totem` определяет target rule и какой эффект запросить.
6. `Totem` возвращает в `Shelf` запрос на применение.
7. `Shelf` выбирает конкретную цель и помещает эффект в event loop.
8. `Shelf` применяет modifier или instant effect к цели только в `APPLY` фазе следующего тика.

Следствие:
- взаимодействие идет через `Shelf`,
- нет прямых actor -> actor вызовов,
- локальная координация остается в рамках одной `Shelf`,
- modifier и instant effect не применяются немедленно actor-источником,
- события, созданные в текущем тике, не оказывают эффекта в этом же тике,
- внутри текущего тика нет немедленного применения эффекта.

## Пример: Metronome + Haste

Сетка:
- slot 1: `Plant A`
- slot 2: `Metronome`
- slot 3: `Plant B`

Flow:
- `Plant A` активируется,
- `Shelf` рассылает локальное событие,
- `Metronome` задает target rule: зеркальная цель по другую сторону,
- `Metronome` создает запрос: применить `haste x2 for 1s`,
- `Shelf` определяет конкретную цель: `Plant B`,
- `Shelf` ставит применение эффекта в свой event loop,
- `Shelf` применяет modifier к `Plant B` в `APPLY` фазе следующего тика,
- `Plant B` ускоряется на следующих тиках.

Важно:
- эффект не обязан проявляться в ту же микросекунду,
- допустима задержка до следующего tick.

## Instant Effects

`Instant effect`:
- проходит через `Shelf`,
- не исполняется напрямую одним actor на другом actor,
- подчиняется тем же ограничениям event loop, что и modifier.

Следствие:
- instant effect, созданный в тике `N`, не применяется немедленно в тике `N`,
- instant effect применяется только через `APPLY` фазу следующего тика.

## Глобальные Modifiers

Допускается внешний слой выше `Shelf`, например `Room`.

Такой слой может:
- применять modifiers ко всем `Shelf`,
- применять modifiers ко всем `Plant` по фильтру.

Примеры:
- ночь -> `value_decrease`
- зима -> `freeze` для части растений

Правило:
- это внешний слой, а не часть локальной логики `Shelf`.

## Теги

У объектов могут быть `tags`.

Теги используются для:
- фильтрации целей,
- условий применения modifiers.

Примеры:
- `plant`
- `totem`
- `wintering`
- `night_sensitive`

Правила:
- теги не заменяют modifiers,
- теги помогают выбирать цели и проверять условия применения.

## Do Not

Не делать:
- modifiers, чье имя зависит от source, например `cat_debuff`,
- прямые вызовы между actor-объектами,
- глобальный event bus,
- modifiers, которые навсегда напрямую мутируют состояние объекта,
- сложную ECS-подобную систему modifiers,
- stacking logic на этом этапе.

## Source Of Truth

Для текущего этапа source of truth такой:
- modifier = длительный эффект на цель,
- aura = постоянное правило влияния, пока активен источник,
- instant effect = мгновенное действие без длительности,
- source = тот, кто запросил применение эффекта,
- source задает target rule, но не выбирает конкретную цель,
- конкретные цели определяет `Shelf`,
- эффекты применяются через `Shelf` и его deferred event loop, а не actor -> actor,
- active modifiers хранятся на самих целях, а не в `Slot` и не в `Shelf`.
