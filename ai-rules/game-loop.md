# Правила Для Game Loop

## Назначение

Этот файл фиксирует архитектурные правила симуляции для инкрементальной механики проекта.

Цель:
- не оставлять агентам свободу выдумывать core loop,
- удерживать симуляцию в модели дискретного тика,
- не допускать мгновенных бесконечных chain reaction,
- не допускать глобальной event-driven архитектуры для этой механики.

## Текущая Модель Объектов

На текущем этапе допустимы только следующие сущности:
- `Shelf` — контейнер и локальный координатор симуляции для своих слотов.
- `Slot` — тупой контейнер.
- `Pot` — контейнер для `Plant`.
- `Plant` — активный участник симуляции внутри `Pot`.
- `Totem` — активный участник симуляции, занимающий слот целиком.

Правила структуры:
- `Shelf` содержит слоты.
- один `Slot` содержит либо `Pot` с `Plant`, либо `Totem`.
- в текущем дизайне `Slot` не содержит одновременно `Pot` и `Totem`.
- `Accessory` не относится к slot occupancy на этом этапе и не должен подменять `Totem`.
- не вводить новые типы объектов без явного указания.

## Общий Принцип Симуляции

Базовое правило:
- каждая `Shelf` симулируется локально и независимо.

Это означает:
- разные `Shelf` не влияют друг на друга,
- `Shelf` координирует только своих локальных участников,
- `Shelf` не является глобальным event bus для всей игры,
- глобальная координация между всеми полками для этой механики не используется.

## Участники Симуляции

Активные участники симуляции на текущем этапе:
- `Plant`
- `Totem`

Не являются участниками симуляции:
- `Slot`
- `Pot` как самостоятельный actor

Обязательные правила:
- `Slot` не тикает,
- `Slot` не активируется,
- `Slot` не диспатчит события,
- `Slot` не содержит свою собственную логику оркестрации,
- `Shelf` может обходить свои `Slot` и собирать активных участников симуляции.

## Модель Времени

Основа симуляции:
- симуляция строится на дискретном тике `Shelf`.

Текущее рабочее предположение:
- `Shelf` обновляется с шагом около `0.1s`.

Уточнения:
- это текущее архитектурное правило, а не вечный hardcoded геймдизайн-параметр,
- механики должны быть совместимы с дискретным шагом такого порядка,
- нельзя проектировать механику вокруг микро-тиков вроде `0.001s`,
- нельзя требовать симуляции каждой миллисекунды как основы расчета.

## Семантика Одного Тика

Один тик `Shelf` должен рассматриваться как ограниченная фаза обработки, а не как бесконечно разворачивающаяся рекурсия.

Нормальная последовательность тика:
1. `Shelf` запускает tick.
2. `Shelf` обходит локальных участников симуляции.
3. `Shelf` определяет, кто активировался на этом тике.
4. `Shelf` собирает результаты этих активаций.
5. `Shelf` координирует реакции локальных участников, например `Totem`.
6. `Shelf` применяет итоговые изменения состояния.
7. Эти изменения начинают влиять на последующие тики, а не обязаны мгновенно породить бесконечную новую волну реакций в рамках этого же тика.

Intentional design choice:
- это уменьшает визуальный мусор,
- это упрощает дебаг,
- это защищает от event hell,
- это повышает стабильность idle/incremental симуляции.

Главное ограничение:
- внутри одного тика нельзя допускать мгновенную бесконечную chain reaction recursion.

## События

События допустимы только как локальный механизм внутри одной `Shelf`.

Правила:
- actor не должен напрямую триггерить всех остальных участников,
- actor сообщает `Shelf` о факте своей активации,
- `Shelf` координирует обработку реакций внутри своей полки,
- локальные реакции не должны требовать глобальной шины,
- для этой механики не использовать global event bus,
- для этой механики не использовать Kafka-like архитектуру.

## Пример С Метрононом

Пример раскладки:
- слот 1: `Pot` + `Plant A`
- слот 2: `Totem Metronome`
- слот 3: `Pot` + `Plant B`

Пример поведения:
- `Plant A` и `Plant B` дают по `1 coin` раз в `10s`,
- когда один из них активируется, `Metronome` может применить ускорение или `charge` к зеркальному объекту по другую сторону от себя,
- при дискретном тике `Shelf` это изменение не обязано мгновенно запустить бесконечную новую цепочку в том же тике,
- допустимо и ожидаемо, что эффект реакции фактически проявится на следующем тике, примерно через `0.1s`,
- такое поведение считается правильным и желаемым.

## Правила Для Charge / Time Push

Допустимы механики, которые:
- уменьшают оставшееся время до активации,
- добавляют `charge`,
- делают `push timer`,
- временно ускоряют достижение следующей активации.

Важно:
- даже если такие механики образуют циклы между объектами, система не должна ломаться,
- циклы допустимы как часть билдостроения,
- но их обработка должна оставаться ограниченной дискретным тиком,
- циклы не должны реализовываться как мгновенная бесконечная рекурсия в рамках одного тика.

## Требования К Стабильности

Система должна:
- быть стабильной в долгой idle-симуляции,
- не деградировать из-за мгновенных бесконечных chain reaction,
- быть предсказуемой,
- быть удобной для отладки,
- масштабироваться как минимум на ориентир порядка `~10 Shelf` по `~12 Slot` каждая.

Уточнение:
- это не hard limit, а ориентир для архитектурных решений.

## Do Not

Не делать:
- глобальный event bus для game loop механики,
- Kafka-подобную систему доставки событий,
- `_process()` на каждом `Plant` или `Totem` как основу симуляции,
- мгновенную бесконечную рекурсию реакций,
- симуляцию на микро-тиках как обязательную основу архитектуры,
- новые сущности или механику без явного указания,
- переименование `Totem` обратно в `Accessory`,
- `Slot` как умный объект симуляции,
- предположение, что один `Slot` одновременно содержит `Pot` и `Totem`.

## Source Of Truth

Для текущего этапа source of truth такой:
- core simulation loop = дискретный tick на уровне `Shelf`,
- `Shelf` координирует только локальных участников своей полки,
- активные actors = `Plant` и `Totem`,
- реакции ограничены рамками тика и не обязаны мгновенно разгонять новую волну реакций в тот же кадр или тот же tick.
