# Примеры Для Game Loop

Этот файл вторичен. Главный source of truth находится в `ai-rules/gameplay/game-loop.md`.

## Пример Одного Тика

Ситуация:
- `Shelf` имеет несколько слотов.
- один `Plant` готов активироваться.
- рядом есть `Totem`, который реагирует на активацию.

Нормальная обработка:
1. `Shelf` запускает tick.
2. `Plant` помечается как активировавшийся на этом тике.
3. `Shelf` фиксирует результат активации.
4. `Shelf` вызывает локальную реакцию `Totem`.
5. `Shelf` применяет изменения состояния.
6. Следующее влияние этих изменений допускается на следующем tick.

## Пример Metronome

Ситуация:
- слот 1: `Pot` + `Plant A`
- слот 2: `Totem Metronome`
- слот 3: `Pot` + `Plant B`

Поведение:
- `Plant A` сработал и дал `1 coin`.
- `Metronome` добавил `charge` для `Plant B`.
- если после этого `Plant B` должен активироваться почти сразу, допустимо, что это будет видно на следующем tick, а не в бесконечной мгновенной цепочке в тот же tick.

## Неправильная Модель

Так работать не должно:
1. `Plant A` активируется.
2. `Totem` мгновенно толкает `Plant B`.
3. `Plant B` мгновенно активируется в той же рекурсии.
4. Это снова триггерит `Totem`.
5. Цепочка продолжается без естественной границы тика.

Это запрещено, потому что ведет к event hell, нестабильности и плохой предсказуемости.

## Допустимая Задержка

Если реакция проявилась на следующем tick, например через `~0.1s`, это считается нормальным поведением, а не багом.
