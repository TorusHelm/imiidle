# Правила Для Shelf Event Loop

## Назначение

Этот документ фиксирует строгий архитектурный контракт event loop для `Shelf`.

Это не реализация и не гайд. Это source of truth для порядка тика, очереди событий и deferred-обработки.

## Роль Shelf

`Shelf` — orchestrator локальной симуляции.

`Shelf`:
- является единственным источником времени для своей полки,
- управляет дискретным tick,
- управляет порядком выполнения фаз,
- маршрутизирует события между участниками симуляции,
- координирует взаимодействия только внутри своей полки.

`Shelf` не является глобальным event bus.

## Модель Времени

Симуляция `Shelf`:
- дискретная,
- работает шагом порядка `~0.1s`,
- обновляет всех локальных actors синхронно,
- не допускает независимых таймеров внутри системы.

Следствие:
- время централизовано в `Shelf`,
- `Plant` и `Totem` не живут в собственной временной модели.

## Очередь Событий

В `Shelf` существует одна общая очередь событий.

Правила очереди:
- очередь работает по принципу `FIFO`,
- приоритетов у событий нет,
- оптимизаций очереди нет,
- дедупликации нет,
- слияния событий нет,
- порядок добавления считается значимым.

Логическое разделение очереди:
- `incoming` — события, которые становятся видны в текущем тике,
- `outgoing` — события и применения, собранные в текущем тике для следующего.

Правило жизненного цикла:
- `incoming` живет только в рамках текущего тика,
- `outgoing` становится `incoming` на следующем тике,
- очередь не должна разрастаться из-за рекурсии внутри одного тика.

## Фазы Тика

Каждый tick `Shelf` состоит из фиксированных фаз.

### 1. APPLY

В начале тика:
- применяются только эффекты из предыдущего тика,
- эффекты воздействуют на состояние actors,
- gameplay events на этом шаге не создаются.

Жесткое правило:
- это единственный момент тика, когда эффекты реально применяются к состоянию.

### 2. UPDATE

После `APPLY`:
- `Shelf` обходит всех локальных actors в порядке `row -> col`,
- внутри `UPDATE` нет разделения по типам вроде `Plant` и `Totem`,
- actors работают уже с состоянием после `APPLY`,
- actors видят только gameplay events из прошлого тика,
- actors не видят `report` и `request`, созданные в текущем тике,
- actors могут сформировать `report` и максимум один `request` за тик.

Дополнительные правила:
- любой actor может иметь максимум одну activation за тик,
- overflow сохраняется,
- дополнительные activation происходят только в следующих тиках.

### 3. COLLECT

После `UPDATE`:
- `Shelf` собирает все `report` и `request`, созданные actors в этом тике,
- порядок сбора определяется обходом `row -> col`,
- собранные данные записываются в `outgoing`.

### 4. BUILD NEXT QUEUE

После `COLLECT`:
- `Shelf` преобразует `report` в gameplay events для следующего тика,
- `Shelf` выполняет target selection для `request` в этом же тике,
- `Shelf` формирует очередь следующего тика,
- `Shelf` не применяет эффекты к состоянию actors на этом шаге.

Жесткие правила:
- gameplay events, созданные из `report` в `N`, становятся доступны только в `N+1`,
- эффекты из `request`, созданных в `N`, применяются только в `APPLY` фазе `N+1`,
- если actor реагирует на gameplay event в `N+1` и создает новый `request`, этот эффект применяется только в `N+2`.

## Deferred Поведение

В системе используется deferred-модель событий.

Это означает:
- событие или эффект, созданный в тике `N`, не оказывает эффекта в тике `N`,
- gameplay events из `N` видны только в `N+1`,
- эффекты, созданные реакцией на события в `N+1`, применяются только в `N+2`,
- любое изменение имеет минимальную задержку в один tick,
- реакция на activation и последующее применение эффекта растянуты по тикам.

Это намеренное поведение системы.

## Генерация Событий

Правила генерации:
- одно событие может породить несколько новых событий,
- actors могут создавать `report`,
- actors могут создавать максимум один `request` за тик,
- обработка текущего тика может приводить к формированию новых gameplay events и будущих применений для следующего тика.

Главное правило:
- все новые события попадают только в очередь следующего тика,
- новые события и эффекты не применяются немедленно.

## Отсутствие Рекурсии

Внутри одного тика:
- события не запускают реактивные цепочки эффектов,
- эффекты не порождают немедленное повторное применение,
- невозможно зациклить систему мгновенной рекурсией.

Следствие:
- `N`: activation,
- `N+1`: reaction,
- `N+2`: apply,
- система остается предсказуемой и ограниченной по фазам.

## Порядок Выполнения

Порядок определяется обходом `row -> col`.

Правила:
- используется единый порядок по слотам,
- события и `request` упорядочиваются этим обходом,
- этот порядок влияет на результат,
- `Shelf` не должен менять порядок событий произвольно.

## Context-Слои

Следующие слои не используют event queue:
- aura,
- влияние `Pot`,
- влияние `Room`.

Правила:
- они не генерируют события,
- они не применяются через `APPLY`,
- они учитываются как постоянный контекст во время `UPDATE`.

## Связь С Plant И Totem

Правила:
- `Plant` и `Totem` обновляются только в `UPDATE` фазе `Shelf`,
- они не имеют собственного времени вне `Shelf`,
- они видят только gameplay events прошлого тика,
- их эффекты приходят только через `APPLY` фазу будущего тика.

## Shelf Не Должен

`Shelf` не должен:
- применять события в середине тика,
- применять события в конце тика,
- вызывать actors вне фиксированных фаз,
- допускать немедленные реакции в том же тике,
- менять порядок событий,
- вводить приоритеты событий,
- оптимизировать очередь через слияние,
- оптимизировать очередь через удаление дубликатов,
- обходить deferred-модель,
- допускать прямые actor -> actor взаимодействия.

## Source Of Truth

Для текущего этапа source of truth такой:
- `Shelf` ведет симуляцию дискретными тиками,
- обход actors идет в порядке `row -> col`,
- в системе одна общая очередь событий на `Shelf`,
- gameplay events из `N` становятся видны только в `N+1`,
- реакции на события происходят не в том же тике,
- эффекты применяются только в `APPLY`,
- типовая схема растянута как `N: activation`, `N+1: reaction`, `N+2: apply`.
