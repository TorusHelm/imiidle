# Контракт Для Visual Composition And Layout

## Назначение

Этот документ фиксирует правила построения визуального слоя, layout и editor workflow.

Это строгий архитектурный контракт для агентов, а не гайд по реализации.

## Назначение Визуального Слоя

Визуальный слой:
- отображает gameplay-состояние,
- не является source of truth для логики,
- должен быть прозрачным,
- должен быть редактируемым,
- должен быть удобным для дебага.

## Разделение Gameplay И Visual

Жесткие правила:
- gameplay layout в терминах `row` / `col` / `index` определяет структуру,
- visual layout строится на основе gameplay-структуры,
- визуальные координаты не используются для gameplay-логики,
- соседство и взаимодействие нельзя определять через пиксельные позиции.

## Иерархия Визуальной Композиции

Визуальная структура должна отражать gameplay-структуру:
- `Room`
- `Shelf`
- `Slot`
- `Pot` + `Plant`
- или `Totem`

Правила:
- структура должна быть читаемой в редакторе,
- вложенность должна быть понятной без анализа кода.

## Editor-Friendly Подход

Жесткие правила:
- сцены должны быть пригодны для просмотра и настройки прямо в редакторе,
- должна быть возможность подставлять разные ресурсы `Shelf`, `Pot`, `Plant`, `Totem` без запуска игры,
- композиция должна быть проверяема визуально.

## Live-Обновление В Редакторе

Жесткие правила:
- изменения в ресурсах, конфигурациях и definitions должны применяться в редакторе без запуска игры,
- визуальный слой должен поддерживать обновление на лету,
- для этого допускается использование editor-time механизмов, например `@tool`.

Важно:
- изменение параметров должно сразу отражаться в preview,
- это является частью рабочего процесса, а не дополнительной функцией.

## Resource-Driven Layout

Все параметры layout задаются через:
- `.tres` ресурсы,
- конфиги,
- definition-объекты.

Запрещено:
- использовать захардкоженные значения в коде,
- вручную задавать offsets и размеры внутри скриптов.

Примеры параметров:
- размеры `Slot`,
- отступы,
- grid-параметры,
- anchor points,
- offsets,
- texture size,
- baseline.

## Базовая Единица: Slot

`Slot` задает базовую область размещения, то есть visual footprint.

Правило:
- все остальные элементы выстраиваются относительно `Slot`.

## Pot И Plant В Визуальной Системе

`Pot` определяет точку привязки для `Plant`.

`Plant` размещается относительно `Pot` через:
- anchor,
- offset,
- view area.

Правила:
- `Pot` и `Plant` логически вписываются в область `Slot`,
- допускается визуальный выход за границы `Slot`, если это сделано намеренно,
- принудительное clipping не используется, если оно явно не требуется.

## Totem В Визуальной Системе

`Totem` размещается в `Slot` аналогично `Pot`.

Правила:
- `Totem` использует тот же базовый footprint `Slot`,
- визуальная посадка `Totem` должна быть консистентной с `Pot`.

## Консистентность Размеров

Жесткие правила:
- все элементы должны подчиняться общей системе размеров,
- нельзя делать произвольные размеры на глаз,
- размеры и пропорции должны быть предсказуемыми.

Важно:
- `Slot` задает основу,
- `Pot` и `Totem` используют один класс размещения,
- `Plant` вписывается в систему через `Pot`.

## Grid И Читаемость

Правила:
- структура `row` / `col` должна быть визуально понятна,
- расположение `Slot` должно соответствовать gameplay-сетке,
- порядок элементов должен быть очевиден при просмотре сцены.

## Debug / Preview Режим

Должен существовать способ визуального дебага композиции.

Он должен позволять:
- видеть границы `Slot`,
- видеть `row` / `col` / `index`,
- видеть anchor points и baseline,
- видеть тип occupant: `Pot` или `Totem`,
- видеть вложенность `Plant` внутри `Pot`.

Правила:
- debug-режим не влияет на gameplay,
- debug-отображение может быть включаемым и выключаемым.

## Подмена Контента

Должна быть возможность легко подставлять разные ресурсы.

Можно менять:
- `Pot`,
- `Plant`,
- `Totem`,
- `Shelf`.

Правило:
- это должно работать прямо в редакторе.

## Visual Layer Не Должен

Визуальный слой не должен:
- определять gameplay-логику,
- использовать пиксельные координаты как источник истины,
- содержать захардкоженные layout-значения,
- зависеть от runtime-инициализации для базового отображения,
- скрывать структуру иерархии,
- смешивать actor-логику и визуальное представление.

## Source Of Truth

Для текущего этапа source of truth такой:
- visual layer отображает gameplay-структуру, но не определяет ее,
- gameplay layout задается через `row` / `col` / `index`, а visual layer только строится поверх него,
- layout должен быть resource-driven и editor-friendly,
- `Slot` является базовой visual unit,
- `Pot`, `Plant` и `Totem` должны вписываться в общую систему размеров и привязок,
- визуальный слой обязан поддерживать preview, дебаг и подмену контента без запуска игры.
