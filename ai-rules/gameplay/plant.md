# Контракт Для Plant

## Назначение

Этот документ фиксирует runtime-поведение и архитектурную роль `Plant`.

Это контракт для агентов, а не гайд по реализации.

## Роль Plant

`Plant` — локальный actor с таймерным циклом.

`Plant`:
- накапливает внутренний progress,
- двигается к своей activation,
- сообщает о своей activation в `Shelf`,
- не управляет другими сущностями,
- не принимает глобальных решений.

## Границы Ответственности

`Plant` отвечает за:
- свое внутреннее состояние,
- свой progress,
- свой age,
- применение полученных эффектов к себе,
- генерацию activation report.

`Plant` не отвечает за:
- маршрутизацию эффектов,
- выбор целей,
- знание других сущностей,
- координацию событий,
- принятие решений за пределами собственного состояния.

Все взаимодействия координируются через `Shelf`.

## Модель Времени

`Plant` не управляет временем самостоятельно.

Жесткие правила:
- `Plant` не использует внутренние таймеры движка,
- `Plant` не использует корутины,
- `Plant` не использует async-логику,
- `Plant` обновляется только через tick от `Shelf`,
- вся временная логика `Plant` основана на `delta`, полученном от `Shelf`.

Следствие:
- симуляция централизована,
- внутри `Plant` нет независимого времени.

## Модель Состояния

`Plant` рассматривается как объект с тремя категориями состояния:

### Базовые параметры

Сюда относятся:
- `cycle_time`
- `value`
- другие собственные базовые характеристики растения

### Runtime-состояние

Сюда относятся:
- `progress`
- `age`
- другие краткоживущие runtime-значения самого растения

### Внешние влияния

Сюда относятся:
- timed modifiers
- instant effects, примененные к растению

Правило:
- modifiers являются внешним слоем влияния,
- modifiers не являются частью базовой идентичности `Plant`.

## Контракт Modifiers

Для `Plant` действуют такие правила:
- modifier определяется типом, а не источником,
- один тип modifier рассматривается как один активный modifier,
- stacking по силе не используется,
- повторное применение обновляет длительность и заменяет значение,
- modifiers влияют только на вычисления поведения,
- modifiers не должны менять структуру `Plant`.

`Plant` не должен зависеть от того, кто был source modifier.

## Timed И Instant Эффекты

`Plant` различает два класса внешних воздействий:

### Timed modifiers

Это эффекты с длительностью:
- `haste`
- `slow`
- `freeze`
- другие аналогичные временные влияния

### Instant effects

Это эффекты без длительности:
- `charge`
- мгновенный сброс cooldown
- мгновенное изменение value

Правило:
- timed modifiers и instant effects — разные механики,
- instant effects не превращаются в timed modifiers.

## Activation

`Activation` для `Plant` — это report в `Shelf`.

Жесткие правила:
- `Plant` не применяет награду напрямую,
- `Plant` не распределяет результат activation,
- `Plant` не знает, какие реакции произойдут после activation,
- `Plant` только сообщает факт activation в `Shelf`.

## Взаимодействие Через События

`Plant` получает события только от `Shelf`.

`Plant` отправляет события только в `Shelf`.

Правила:
- `Plant` не общается напрямую с `Totem`,
- `Plant` не общается напрямую с другими `Plant`,
- `Plant` не знает источник события как архитектурную зависимость,
- `Plant` реагирует только на входящие данные и собственное состояние.

Базовый flow:
- `Plant` -> `Shelf`: report
- `Shelf` -> `Plant`: event
- `Plant` -> `Shelf`: request

## Детерминизм

Поведение `Plant` должно быть детерминированным.

Правило:
- одинаковый внутренний state,
- одинаковые входящие события,
- одинаковый tick/delta
= одинаковый результат.

`Plant` не должен зависеть от скрытого порядка побочных эффектов или внешней недетерминированной логики.

## Жесткие Ограничения

`Plant` не должен:
- напрямую взаимодействовать с другими сущностями,
- хранить ссылки на соседей,
- хранить ссылки на `Slot`,
- выбирать цели эффектов,
- маршрутизировать события,
- использовать внутренние таймеры,
- использовать async-логику,
- зависеть от source modifier,
- обходить `Shelf`,
- принимать на себя роль orchestrator.

## Source Of Truth

Для текущего этапа source of truth такой:
- `Plant` — локальный actor со своим progress-циклом,
- время приходит в `Plant` только через tick `Shelf`,
- `Plant` сообщает activation в `Shelf`, но не обрабатывает последствия сам,
- внешние эффекты применяются к `Plant`, но координируются не им,
- `Plant` не знает других участников симуляции и не взаимодействует с ними напрямую.
