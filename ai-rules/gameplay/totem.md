# Контракт Для Totem

## Назначение

Этот документ фиксирует runtime-поведение и архитектурную роль `Totem`.

Это строгий контракт для агентов, а не гайд по реализации.

## Роль Totem

`Totem` — это actor.

`Totem`:
- реагирует на gameplay events,
- может накапливать собственный progress,
- может иметь условную activation,
- не является time-based actor в модели `Plant`.

Activation `Totem` определяется не временем, а условиями.

Источники активации могут быть такими:
- накопление событий,
- реакция на gameplay events,
- постоянное влияние через ауру.

## Границы Ответственности

`Totem` отвечает за:
- хранение собственного локального состояния,
- обработку входящих gameplay events,
- определение момента своей activation,
- генерацию `report` и `request`.

`Totem` не отвечает за:
- выбор конкретных целей,
- прямое применение эффектов,
- знание других сущностей,
- маршрутизацию событий,
- координацию симуляции.

Все взаимодействия координируются через `Shelf`.

## Модель Активации

`Totem` может накапливать progress, например `count`.

Правила:
- activation происходит при достижении threshold,
- overflow сохраняется,
- за один tick допускается не более одной activation,
- если накоплено больше, дополнительные activation происходят в следующих тиках.

Следствие:
- даже при большом входном накоплении `Totem` не должен разворачивать несколько activation в одном tick.

## Работа С Событиями

`Totem` реагирует только на gameplay events.

Правила:
- `Totem` видит только gameplay events прошлого тика,
- `Totem` не видит события, созданные в текущем тике,
- служебные события `Shelf` не являются валидными trigger,
- `Totem` может реагировать на все gameplay events,
- `Totem` может использовать whitelist валидных типов событий,
- решение о реакции основано только на входном событии прошлого тика и локальном состоянии `Totem`.

## Ограничение На Самовлияние

Жесткие правила:
- `Totem` не реагирует на события, source которых является он сам,
- `Totem` не может быть целью собственных эффектов,
- собственные activation и effects не считаются валидным input для самого `Totem`.

Это исключает самозапуск и self-loop поведение.

## Генерация Эффектов

При activation `Totem` не применяет эффекты напрямую.

`Totem`:
- формирует `report` или максимум один `request` за тик,
- передает его в `Shelf`,
- не изменяет состояние других actors напрямую.

`Shelf`:
- принимает запрос,
- определяет дальнейшее применение,
- координирует эффект в рамках общей event-модели.

## Targeting

`Totem` может задавать только target rule.

Примеры target rule:
- соседние элементы,
- все на `Shelf`,
- случайный валидный,
- по типу,
- по условию.

Жесткие правила:
- `Totem` не выбирает конкретные цели напрямую,
- конкретные цели выбирает `Shelf`,
- любая случайность реализуется только на стороне `Shelf`.

## Аура

Аура — это отдельный механизм постоянного влияния.

Правила:
- аура не связана с event trigger моделью,
- аура не является timed modifier,
- аура не генерирует события каждый tick,
- аура не использует event queue,
- аура действует как постоянное правило, пока `Totem` активен,
- аура учитывается в `UPDATE` как часть контекста.

Следствие:
- `Shelf` учитывает активные ауры как часть симуляции,
- аура не должна имитироваться через постоянный спам событий.

## Локальное Состояние Totem

`Totem` может хранить только собственное состояние, нужное для своей логики.

Допустимые примеры:
- накопленный `progress` или `count`,
- внутренние флаги,
- состояние activation,
- собственные modifiers, если они ему нужны.

`Totem` не должен хранить:
- ссылки на другие сущности,
- глобальное состояние `Shelf`,
- список целей,
- историю всех событий системы.

## Позиция На Shelf

Позиция `Totem` на `Shelf` имеет gameplay-значение.

Правила:
- порядок элементов влияет на результат,
- соседство является частью механики,
- adjacency считается осознанным элементом билдостроения.

При этом:
- само вычисление конкретных целей и соседей остается ответственностью `Shelf`,
- `Totem` не должен хранить прямые ссылки на соседние сущности.

## Детерминизм

Поведение `Totem` должно быть детерминированным.

Правило:
- одинаковое локальное состояние,
- одинаковые входные события,
- одинаковый порядок событий
= одинаковый результат.

Случайность:
- не живет внутри `Totem`,
- централизуется через `Shelf`.

## Totem Не Должен

`Totem` не должен:
- напрямую взаимодействовать с другими сущностями,
- выбирать конкретные цели самостоятельно,
- применять эффекты напрямую,
- реагировать на собственные события,
- использовать внутренние таймеры, не связанные с событиями,
- генерировать бесконечные цепочки в рамках одного tick,
- зависеть от служебных событий системы,
- обходить `Shelf`,
- принимать на себя роль orchestrator.

## Source Of Truth

Для текущего этапа source of truth такой:
- `Totem` — actor, который реагирует на gameplay events и условия,
- activation `Totem` не time-based по модели `Plant`,
- `Totem` хранит только собственное локальное состояние,
- `Totem` может иметь максимум одну activation за тик,
- `Totem` видит только gameplay events прошлого тика,
- `Totem` отправляет `report` и максимум один `request` в `Shelf`, но не применяет эффекты напрямую,
- target rule задается `Totem`, но конкретные цели определяет `Shelf`,
- self-trigger и self-target для `Totem` запрещены.
